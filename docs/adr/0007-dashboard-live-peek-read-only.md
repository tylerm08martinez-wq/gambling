# Dashboard Live Peek Is a Read-Only, Non-Authoritative Second Grader

The dashboard's Pick Log "check this bet" control (🔍) may compute a [[Live Peek]] — a client-side grade of an unsettled [[Pick]] — but that Peek is **read-only and never persisted**. The nightly auto-resolver (ADR 0004) remains the **sole writer** of any settled outcome. If a Peek and the official resolution disagree, the resolution wins.

**Context:** The 🔍 control historically always opened a generic Google search ("MLB Jonathan Aranda Total Bases vs Tigers score 2026-06-01"), forcing the bettor to hunt through results — useless for a single player's stat line and redundant for a [[Game-Line Bet]] whose `final_score` is already stored on the pick. We want 🔍 to "show exactly what I'm looking for cleanly." The clean answer for an unsettled MLB/NBA bet is to call the same public APIs the resolver uses ([[Player Prop Source]]: MLB Stats API, ESPN) and grade it in the browser. But grading a money-scoring bet in client-side JS — separate from the Python resolver that is the project's single test surface — creates a second code path that can produce an outcome. This project's worst bug class is silent mis-resolution (a lost bet scored a win, CLV against the wrong line), and any change touching resolution/scoring/CLV must pass `/verify` against a real finished game — something browser JS cannot easily do.

**Considered Options:** (a) keep the Google search, just craft a better query; (b) let the Peek **write back** a confident grade into `picks.json` via the GitHub token, the way My Bets persists — faster settlement; (c) compute the Peek but treat it as a disposable, clearly-labeled read-only preview that never touches the pick record.

**Decision:** Option (c). The Live Peek is a convenience read, not a settlement action. It is labeled in the UI as a peek ("official grade posts tonight"), re-fetched fresh on each view, and **never written to `picks.json`**. Only sports with a Player Prop Source adapter (MLB, NBA) produce a Peek; every other sport and any fetch failure falls back to an external deep-link (StatMuse for a [[Player Prop]], ESPN scoreboard for a Game-Line Bet). The Peek reuses the resolver's stat-keyword → statKey mapping (ADR 0004) rather than inventing its own — e.g. total bases reads `batting.totalBases` directly from the boxscore, it is not re-derived from hit components — but the mapping lives in two languages now, so the JS is kept deliberately thin and the Python resolver stays canonical.

**Consequences:** The bettor gets a clean, immediate answer for the dominant MLB/NBA case without waiting for the nightly run, and a wrong Peek (postponed game, stat correction, parse miss) costs nothing because it is never persisted and never affects P&L, ROI, or CLV. The cost is a second grading path that can drift from the resolver; this is contained by making the Peek non-authoritative and read-only, so drift surfaces as a visibly-wrong disposable number rather than a corrupted result. Reintroducing write-back later would re-open the dual-writer mis-resolution surface this decision exists to keep closed — a future change that wires the Peek to `picks.json` must revisit this ADR and carry the `/verify` burden the resolver carries.
