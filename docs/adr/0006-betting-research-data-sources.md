# Run the Research Routine From Residential Egress, Not the Cloud

The daily pick-generation routine (V1-Trends + V2-Sharp research) must run from a **residential-IP host** (Tyler's home PC), not Anthropic's cloud. The skills and their HTML data sources (cleatz splits, vegasinsider line movement, oddstrader/bettingpros props) are **left unchanged** — the only thing that moves is *where the routine executes*.

**Context:** A confirmed A/B test on 2026-06-01 showed every research scrape source returning **HTTP 403** to the cloud routine's datacenter egress IP while returning **200 + full data** to a residential IP at the same minute. This is WAF/bot-blocking on the cloud egress, not stale URLs — so the skills' "swap the stale URL" self-healing could never fix it (every scrape from the blocked IP 403s regardless of target). V2-Sharp had sat out 15+ consecutive sessions for this reason. The root cause is the **IP**, not the sources.

**Considered Options:**
- **The Odds API (paid).** Authenticated odds API, datacenter-IP-tolerant. Would supply cross-book props + historical snapshots — but requires a **paid plan** and a routine secret, and still sells **no public ticket/handle splits** (so Hard/Soft RLM would die cloud-side and Steam would have to be re-derived from historical snapshots). Rejected: Tyler declined the paid key, and it forces a large data-layer rewrite plus the loss of RLM for a signal the residential path keeps for free.
- **Residential proxy on the cloud routine.** Keeps the unattended cloud run, routes fetches through an unblocked IP. Keeps all signals, no rewrite — but ongoing proxy cost and reintroduces a routine secret (the proxy creds), the exact thing ADR 0005 avoided. Rejected as primary: pays monthly to rent the residential IP Tyler already owns.
- **Free-tier Odds API + ESPN, game lines only.** $0, stays cloud — but drops **player props** (the documented Primary Edge) and Steam entirely. Rejected: guts the project's core.
- **Run from residential egress — Tyler's home PC (chosen).** $0, **zero data-source rewrite**, no API key, no secret. Every existing scraper already returns 200 from the home IP, so props + RLM + steam all keep working exactly as written.

**Decision:** Move the 9am-Arizona routine off the Anthropic cloud trigger and onto a scheduler on Tyler's **home PC** (Windows Task Scheduler), which runs `claude` headless against the cloned repo with the same prompt the cloud routine used, then commits/pushes picks and posts to #bet-picks. The PC was chosen over the Mac because it is the host Tyler keeps available. Because the PC is on a residential connection, the existing HTML sources are reachable, so **no skill code and no data sources change** — this ADR supersedes nothing in the skills themselves; it relocates execution. The "Self-Healing (swap stale URL + commit)" protocol in `sports-betting-sharp/SKILL.md` becomes a no-op in practice (sources no longer 403) but is left as a safety net.

**Acceptance criterion:** the routine fires from the PC at 9am AZ and #bet-picks shows the feeds returning **200 + real lines/props** (reproducing the 2026-06-01 A/B test from the unblocked IP), with picks logged and pushed.

**Consequences:** Props, splits, RLM, and steam all work again with no rewrite and no recurring cost — the largest possible simplification (we delete a planned migration instead of building it). The cost is the loss of the "PC does not need to be on" property: the home PC must be powered and awake at ~9am AZ (mitigable with a scheduled wake timer). Setup work moves to host configuration — Task Scheduler job, a wake timer, local `git` push auth, Slack MCP availability, and ensuring the skills' `python3` invocation resolves on Windows (`python3`/`py` shim) — not betting logic. Reliability now depends on the home PC rather than Anthropic's infra; a missed day is a powered-off PC, recoverable by a manual run. CLAUDE.md's "runs in Anthropic's cloud" automation note must be updated to reflect the PC host.
