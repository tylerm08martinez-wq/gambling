# Betting Research Sources: BettingPros API Data Layer, Run From Residential Egress

The pick-generation skills (`sports-betting` V1-Trends, `sports-betting-sharp` V2-Sharp) source player props, cross-book odds, and game/event data from the **BettingPros API** (`api.bettingpros.com`) — structured JSON, replacing brittle HTML scraping — and the routine runs from a **residential-IP host (Tyler's home PC), not the Anthropic cloud.** Originally this ADR chose BettingPros to keep the routine *unattended in the cloud* at $0, on the inference that its API Gateway gated on key not IP; the 2026-06-01 cloud acceptance test **disproved that** (403 from the datacenter IP), so execution moved to residential egress where the same client returns 200. See the ACCEPTANCE RESULT below for the full disproof and the no-free-cloud-source finding.

**Context:** A confirmed A/B test on 2026-06-01 showed every HTML scrape source (cleatz public splits, vegasinsider line movement, oddstrader/bettingpros *web* pages) returning **HTTP 403** to the routine's datacenter egress IP while a residential IP got **200 + full data** at the same minute — IP/WAF bot-blocking on the cloud egress, not stale URLs. V2-Sharp had sat out 15+ consecutive sessions for this reason. Player Props are the project's documented Primary Edge, so the research path cannot drop them. Investigation (three research agents, 2026-06-01) established three things: (1) header/UA/fingerprint tricks cannot defeat an IP-reputation block; (2) a headless browser does **not** help — the block is IP-based so browser realism is irrelevant from a datacenter IP, the sites are server-rendered so a browser adds nothing anyway, and the routine sandbox cannot practically run Chromium; (3) BettingPros exposes a **public-key JSON API** whose missing-key response is `MissingAuthenticationTokenException` (an auth error, not an IP block), meaning it serves datacenter IPs.

**Considered Options:**
- **Run from residential egress (home PC).** $0, zero data-layer rewrite, keeps every existing source including cleatz splits/RLM. Rejected as primary: loses the "PC does not need to be on" property — the home PC must be powered and awake at ~9am AZ — to rent an outcome the BettingPros API gives in the cloud for free. Kept as fallback.
- **The Odds API (paid).** Cleanest single integration with historical snapshots, but needs a paid plan and a routine secret, and still sells no public splits. Rejected: Tyler declined the paid key.
- **Residential proxy on the cloud routine.** Keeps the cloud run, no rewrite — but ongoing proxy cost and reintroduces a routine secret, to rent the residential IP Tyler already owns. Rejected.
- **Header/UA tricks or a headless browser in the cloud.** Rejected: cannot defeat IP-reputation blocking; sources are server-rendered; the sandbox can't run a browser. (Recorded so it is not re-attempted.)
- **BettingPros key-less API as the cloud data layer (chosen).** Datacenter-tolerant, $0, no paid key, restores the Primary Edge.

**Decision:** Rewrite the skills' research data layer to read from `https://api.bettingpros.com/v3/` (reusing the shared HTTP client / browser `User-Agent` + backoff, mirroring ADR 0005's ESPN path). Confirmed-working endpoints (verified from residential 2026-06-01; datacenter-tolerance to be proven by a cloud re-trigger): `/events?sport=MLB&date=…` (games with `scheduled` UTC time + probable pitchers), `/props?sport=MLB` (player props across multiple `book_id`s — cross-book gaps), `/offers?market_id=…&event_id=…` (cross-book odds). The public `x-api-key` is **scraped live from BettingPros' site JS at run time** (static keys can rotate), never committed. On any BettingPros API failure the Scheduled Run **edits and commits nothing**, logs zero picks, and posts a specific `"BettingPros API unavailable: <reason>"` to #bet-picks — this retires the dead "Self-Healing (swap stale URL + commit)" protocol that the routine's no-non-tracker-commit contract (ADR 0003) made unexecutable. `cleatz` public splits remain Cloudflare IP-reputation-blocked from the cloud, so **Hard/Soft RLM are sourceable only on a Manual Run** unless the PRD confirms BettingPros also exposes usable public consensus/ticket data.

**Open items — RESOLVED by the slice-3 (#51) live probe, 2026-06-01:**
1. **Public splits — NOT available.** The BettingPros API exposes consensus *pricing* and a `pm_win_pct` probability, but no ticket%/handle% splits. So **Hard/Soft RLM are Manual-Run-only** (CONTEXT.md updated); a Scheduled Run's sharp game-line edge is Steam.
2. **Line movement — available.** Each `/offers` selection carries an `opening_line` plus every book's current line/odds. [[Steam Move]] is operationalized as **3+ books currently moved off their opening line in the same direction** (no free historical snapshots, so this opening-vs-current form replaces the abandoned 5-min-window idea).
3. **Pinnacle — present (`book_id` 2).** The pre-log CLV check can de-vig Pinnacle from `/offers`. Automated Measured CLV at resolution stays a separate `/verify`-gated follow-up.
4. Fold in the audited skill-correctness fixes (independent of data source): retire self-heal-and-commit → fail-loud-to-Slack (P0); fix the game-time DST mislabeling, now mostly mooted by BettingPros' UTC `scheduled` field (P0); gate scheduled V1 to a Market-Confirmed Primary Edge so `ats_trend`/consensus-only candidates become rejected candidates, not logged picks (P1, ADR 0003); reconcile V1's conflicting score thresholds (P1); add an explicit signal→canonical `primary_edge_type` mapping per skill (P1).

**Acceptance criterion:** a live cloud re-trigger of the routine shows the BettingPros endpoints returning **200 + real props/odds** in #bet-picks with picks logged and pushed. **If the cloud re-trigger 403s, fall back to the home-PC residential path.**

**ACCEPTANCE RESULT (2026-06-01) — FAILED, pivoted to home-PC.** The cloud re-trigger returned **HTTP 403 from BettingPros across MLB + NBA, props + events** — including the keyless `/props`. The earlier "API Gateway gates on key, not IP" inference was drawn from a *residential* IP (where the only error was a missing-key auth error) and never tested the datacenter egress; BettingPros' CloudFront **does IP-reputation-block the Anthropic datacenter range**, same as cleatz. A follow-up investigation confirmed **no free cloud-reachable source exists for cross-book player props** (the Primary Edge): the only sources proven reachable from the routine env are ESPN's hidden API and the MLB Stats API, and ESPN exposes only **single-book (DraftKings) game lines** — no cross-book grid, no props, no public splits. The Odds API (paid) was declined.

**Decision (revised): run the routine from a residential-IP host (Tyler's home PC), not the cloud.** The BettingPros client + extractor built for this ADR are **retained as the data layer** — they return 200 from a residential IP, so all of slices 1–5 ship and are used; only the *execution location* moves off the Anthropic cloud trigger. A scheduler on the home PC runs `claude` headless against the cloned repo with the routine prompt, then the skills push picks and post to #bet-picks as before. The cloud trigger is disabled once the home-PC run is verified, to stop the daily 403 sit-out noise.

**Consequences:** The Primary Edge (props) + cross-book gaps + Pinnacle CLV all work again from the residential IP, with the deterministic, tested BettingPros client replacing brittle scraping — a real improvement over the pre-2026 scraping even though execution left the cloud. The cost is the loss of the "PC does not need to be on" property: the home PC must be powered and awake ~9am AZ (mitigable with a scheduled wake timer). The fail-loud behaviour (verified in the failed cloud run: V2 posted "BettingPros API unavailable", logged zero picks, fabricated nothing, committed nothing) means a residential outage degrades safely. Reliability now depends on the home PC rather than Anthropic infra; a missed day is a powered-off PC, recoverable by a manual run. CLAUDE.md's "runs in Anthropic's cloud" note is updated to reflect the home-PC host. The de-vig/scoring/resolution code is untouched, so this change does not require `/verify`.
