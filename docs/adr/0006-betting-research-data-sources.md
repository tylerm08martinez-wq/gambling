# Betting Research Data Sources: The Odds API for Props, ESPN for Game Lines

The pick-generation skills (`sports-betting` V1-Trends, `sports-betting-sharp` V2-Sharp) must source odds, props, and line movement from **datacenter-IP-tolerant structured feeds**, not HTML scraping. Player props and their historical snapshots come from **The Odds API** (authenticated, paid); game lines and line movement come from the key-less **ESPN hidden API** already trusted for resolution. This extends the ADR 0004 / ADR 0005 principle — "the routine's datacenter egress IP 403s on HTML scraping" — from the *resolution* side to the *research* side.

**Context:** A confirmed A/B test on 2026-06-01 showed every research scrape source (cleatz public splits, vegasinsider line movement, oddstrader/bettingpros props) returning **HTTP 403** to the Scheduled Run's datacenter egress IP while returning **200 + full data** to a residential IP at the same minute. This is WAF/bot-blocking on the cloud egress, not stale URLs — so the skills' historical "swap the stale URL" self-healing could never fix it (every scrape from the blocked IP 403s regardless of which site it targets). V2-Sharp had sat out 15+ consecutive sessions for this reason. Player Props are the project's documented Primary Edge, so the research path cannot simply drop them.

**Considered Options:**
- **Stay fully key-less (ESPN only).** No secret, consistent with ADR 0005's anti-secret lean — but ESPN gives only game lines (often single-provider, no cross-book grid, no historical snapshots) and **no player props**. Rejected: it permanently abandons the Primary Edge cloud-side.
- **Residential proxy for the existing scrapers.** Keeps splits/RLM alive, zero skill-logic change — but adds a paid proxy, ongoing credential management, and brittleness as sites tighten WAFs, to preserve a ticket-percentage signal that is inherently noisier than line movement. Rejected as primary.
- **The Odds API for everything.** Cleanest single integration, but needs a key + paid plan and still cannot supply public ticket/handle splits (no odds API does).
- **Hybrid: key-less ESPN for game lines, keyed Odds API only for cross-book props + historical snapshots (chosen).** Scopes the secret to exactly the data nothing else provides.

**Decision:**
1. **Data sources.** Game lines / line movement → **ESPN hidden API** (key-less, reuses the shared HTTP client). Cross-book **player props** and **5-minute historical snapshots** → **The Odds API** (`region=us` for books; `region=eu` checked for Pinnacle). This reverses ADR 0005's "avoid routine secrets" lean *only* for props, because props are the Primary Edge and have no key-less cross-book source.
2. **RLM becomes Manual-Run-only.** No odds API sells Public Ticket Data, and the splits sites that do 403 the routine. So **Hard RLM** and **Soft RLM** cannot be Scheduled-Run Primary Edges (they remain valid taxonomy values for Manual Runs from a residential IP). A Scheduled Run's sharp game-line signal is **Steam Move**, detected by diffing consecutive Odds API historical snapshots — "3+ books simultaneously" is operationalized as "≥3 books each moved ≥threshold in the same direction within one 5-minute snapshot window."
3. **CLV source only, not CLV automation.** This change repoints the skills' *pre-log* CLV check at the Odds API Pinnacle line when present, else "CLV unverified" — identical behavior to today, de-vig math untouched, no scoring change. **Automated Measured CLV at resolution off historical snapshots is an explicit follow-up** (the "resolver hardening" ADR 0003 already names) and must be its own `/verify`-gated change.
4. **Key management + no self-edit.** The key is read from `ODDS_API_KEY` (a routine environment secret for cloud runs, local env for manual), never hardcoded or committed. If the key is absent, props degrade to unavailable and the skills fall back to ESPN game lines, noting it — a missing key never crashes a run. The mandatory **"Self-Healing (swap stale URL + commit)" protocol is retired**: with a single authenticated API there is no URL list to churn, and the only failure modes (`401` bad key, `402/429` quota, `5xx` outage) require a human, not a SKILL edit. On any Odds API failure a Scheduled Run **edits and commits nothing**; it logs zero picks and posts a specific `"Odds API unavailable: <reason>"` to #bet-picks. This resolves the ADR 0003 conflict where the routine prompt forbade the non-tracker commits the self-heal protocol required.

**Acceptance criterion:** a live re-trigger of the routine shows the feeds returning **200 + real lines/props** in #bet-picks (reproducing the 2026-06-01 A/B test with the fix in place) — not merely green unit tests.

**Consequences:** Props and cross-book line shopping become reachable from the cloud routine for the first time since the WAF blocking began; V2 stops sitting out for unreachable data. The costs: one routine secret and a paid Odds API plan (credit budget sized in the PRD; historical snapshots cost 10× standard credits), a dependency on an unofficial ESPN odds endpoint (isolated behind the same client as ADR 0005, degrades to "unavailable" not "wrong"), and the loss of ticket-percentage RLM on Scheduled Runs — accepted because Steam is a purer sharp signal and RLM remains available on Manual Runs. The de-vig/scoring/resolution code is untouched, so this change does not require `/verify`; the live re-trigger is its proof.
