# NBA Player Props Resolve From the ESPN Hidden API

Auto-resolution of NBA [[Player Prop]] bets must read per-player stats from a structured JSON source, the **ESPN hidden API** (`site.api.espn.com`), accessed through the same shared HTTP client (browser `User-Agent` + retry/backoff) used for MLB. This extends ADR 0004 — which built the classify-before-resolve engine for MLB only — to a second sport, without changing the never-guess guarantee.

**Context:** ADR 0004 scoped automated Player Prop resolution to MLB via the MLB Stats API, leaving NBA props (a large share of the Primary Edge) manual. Wiring NBA in requires a structured per-player boxscore source that works from the nightly routine's **datacenter egress IP** — the same constraint that forced ADR 0004 off HTML scraping (those pages 403 from cloud IPs). The candidate sources differ mainly on IP tolerance and operational friction, not data quality.

**Considered Options:**
- **`stats.nba.com`** — the official NBA stats endpoints. Rejected: hardest to use from datacenter IPs (aggressive WAF/403 + strict required-header dance), an awkward `resultSets` (header arrays + row arrays) shape, and exactly the cloud-IP block problem ADR 0004 exists to avoid.
- **NBA CDN liveData (`cdn.nba.com/static/json`)** — official, key-less, CDN-fronted (datacenter-IP friendly). Viable, but requires a separate scoreboard feed for game-ID lookup and an NBA-specific JSON shape.
- **balldontlie (`api.balldontlie.io`)** — the cleanest, most stable documented contract, but now **requires an API key** and is rate-limited (~5 req/min) on the free tier. Rejected as primary: it introduces secret management into the cloud routine and a rate constraint, for no data-quality gain over the chosen source.
- **ESPN hidden API (`site.api.espn.com`)** — free, **no API key**, clean JSON, tolerant of datacenter IPs (CDN-fronted), and the closest structural analog to the MLB Stats API path already in `tracker.py`.

**Decision:** NBA Player Props resolve from the ESPN hidden API. A game lookup finds the final NBA game for the bet's date/team (analogous to `find_mlb_game_for_bet`); the per-game summary/boxscore endpoint (`/apis/site/v2/sports/basketball/nba/summary?event={gameId}`) supplies per-player stats. No API key and no special headers are required; the call reuses the shared `_http_get_json` client (browser `User-Agent` + exponential backoff), so a persistent block leaves the pick open rather than crashing or guessing — identical degradation to MLB.

**Boxscore adaptation contract:** ESPN's response is adapted at the NBA client boundary into the same sport-agnostic per-player shape the resolver already consumes — a record exposing `person.fullName` and a `stats.<group>.<key>` structure — so `resolve_prop_value` and `prop_outcome` stay source-agnostic and shared across both sports. The real ESPN shape (verified against the live endpoint while implementing #24): `summary["boxscore"]["players"]` is a list of two team blocks; each `block["statistics"][0]` carries a **parallel `labels`/`stats` array** (e.g. `labels = ["MIN","PTS",...]`) plus `athletes[].athlete.displayName`. Critically, ESPN's `statistics[].name` is **`null`**, not a group name — so the adapter does **not** key off it; it locates each stat by its **label position** (e.g. the index of `"PTS"`) and emits the value under a **synthetic** `("scoring","points")` group/key that the NBA stat-keyword map points at. DNP players have an empty `stats` array and are dropped from the adapted boxscore, so `resolve_prop_value` returns a reason and the pick is left open — never guessed. The NBA stat-keyword map (points; rebounds, assists, three-pointers made, steals, blocks to follow in #25) lives alongside `PROP_STAT_MAP`; combo props such as points+rebounds+assists are resolved by **summing mapped component stats**, never a separate combo key.

**Consequences:** NBA props — a major slice of the Primary Edge — become auto-resolvable with no secrets and no rate-limit handling in the nightly routine, reusing the MLB resolution machinery and its never-guess hard guard. The cost is a dependency on an **unofficial** ESPN endpoint that could change without notice; because the source is isolated behind the adapter and the resolver degrades to "left open" on any failure, a breaking change surfaces as un-resolved picks (manual fallback), never wrong results. If ESPN proves unstable, the adapter boundary lets us swap to the NBA CDN feeds or balldontlie without touching `resolve_prop_value`/`prop_outcome`. Scope remains MLB + NBA Player Props and Game-Line Bets; other sports and Measured CLV stay out of scope.
