# Auto-Resolve Classifies Bet Type Before Resolving

The nightly auto-resolver must classify every open pick as a [[Game-Line Bet]] or a [[Player Prop]] **before** choosing how to resolve it, and must resolve a Player Prop only from that player's boxscore stat line — never from the final game score. This is the follow-up to ADR 0003, which flagged nightly resolver hardening (especially for props and totals) as required but incomplete.

**Context:** Player Props are 67% of logged picks and the documented Primary Edge, yet `tracker.py auto-resolve` historically resolved only MLB moneyline and run line via game margin. A prop such as "Taj Bradley Over 5.5 Strikeouts vs Pittsburgh Pirates" would, if it reached the resolver, match the opponent team, find the game Final, and resolve as a **moneyline on game margin** — a silently wrong result. Props were therefore resolved by web-searching box-score sites (ESPN, MLB.com, Baseball-Reference, etc.). Those HTML pages increasingly return 403 to datacenter egress IPs, so the cloud routine skipped resolvable picks. The MLB Stats API (`statsapi.mlb.com`) is a structured JSON source that carries per-player stats and does not WAF-block HTML-style; it is the durable resolution source.

**Considered Options:** keep web-search-first and only harden it against 403s; patch the reported 403 with a User-Agent and call it done; move all resolution to a local cron on a residential IP; build automated Player Prop resolution into the MLB Stats API path with a bet-type classifier and a hard no-fall-through guard.

**Decision:** `auto-resolve` classifies each open MLB pick (using the `bet_type` field when present, inferring from bet text otherwise) and routes it: Game-Line Bets (moneyline, run line, game total) resolve from the final score; Player Props resolve from `/game/{gamePk}/boxscore` via a stat-keyword → statKey map (`strikeouts`→`pitching.strikeOuts`, `hits`→`batting.hits`, `total bases`→`batting.totalBases`, `rbi`→`batting.rbi`). An `N+` line means Over N−0.5. Player matching is case/accent-insensitive on last name within the identified game; a same-last-name collision is skipped, not guessed. **Hard guard:** if a pick classified as a Player Prop cannot confidently resolve all of {game, player, stat, side}, it is skipped and left open with a specific reason — it must never fall through to the margin path. Unmapped stat = skip. The statsapi call sends a browser `User-Agent` and retries with backoff; a persistent block leaves the pick open to resolve on the next run. Scope is MLB only; NBA and other props remain web-search/manual until a structured source is wired up.

**Consequences:** the dominant pick category resolves automatically and the silent mis-scoring path is closed permanently. The resolver degrades safely — a blocked API or an unparseable bet produces an open pick, never a fabricated or wrong result, consistent with the project's never-guess rule. Coverage is intentionally partial: NBA props and unmapped MLB stats still require manual resolution, and Measured CLV remains out of scope for this change.
